# Maze 项目需求与设计文档

> 本文档旨在详细描述“迷宫（Maze）”桌面应用的开发目标、功能、UI/UX 设计、技术架构及实现细节，供 AI 开发者作为唯一参考依据。  
> 请 AI 在开发过程中严格遵循本文档，**如有任何需求或技术方案的不确定性，必须主动提问并等待确认，禁止自行假设或过度设计**。

---

## 1. 项目概述

**项目名称**：Maze  
**项目类型**：桌面应用  
**技术栈**：Electron + Vue 3 + TypeScript + PrimeVue  
**核心体验**：生成一个居中显示、由线条墙壁构成的随机迷宫，用户使用方向键控制一个角色（黑点或自定义图标）在迷宫中移动，实时记录并展示耗时、行走长度等信息，支持保存当前迷宫参数以便下次恢复。

**设计理念**：极简、高级、富有设计感，参考 Claude Code、Gemini 等现代 AI 工具界面的视觉风格（暗色主题、柔和阴影、微光效、玻璃态/半透明面板、无边框沉浸感）。

## 2. 功能需求（MVP 范围）

### 2.1 迷宫生成
- 随机生成一个二维迷宫，迷宫由**线条墙壁**构成（墙壁为 1px~2px 的线段，通道为空白区域）。
- 支持设定基本参数：
  - 大小（如行数 × 列数：`rows`、`cols`，默认 15×15）
  - 复杂度/分支度（可通过调整算法随机种子或密度实现）
- 每次打开应用或点击“重新生成”按钮时，生成全新迷宫。
- 迷宫必须有**唯一入口**和**唯一出口**，入口在迷宫外围左上角，出口在右下角（或相反），并在视觉上明确标识。

### 2.2 角色控制与动画
- 角色初始位于迷宫入口处。
- 使用**键盘方向键**（↑↓←→）控制角色移动。
- 角色移动必须**流畅平移**，采用基于 `requestAnimationFrame` 的线性插值动画，帧率稳定，移动速度可配置（默认每秒移动 3 个迷宫单位格）。
- 角色与墙壁的碰撞检测：**角色中心与墙壁线条的精确几何检测**（或基于网格的简化逻辑：角色处于某一单元格，该单元格四个方向若有墙壁则不可通行）。
- 达到出口时触发胜利状态（展示提示，停止计时，并可选择重新开始）。
- 角色外观：默认为一个实心黑色小圆点（直径约为单元格边长的 40%），后续可替换为自定义图标（通过设置项）。

### 2.3 数据统计与显示
- **计时器**：从角色第一次移动开始计时，精确到 0.1 秒，到达出口停止。
- **行走长度**：累计角色实际走过的**像素距离**（计算每一帧位移的欧氏距离累加），不按网格数计算。
- **步数**：角色每完成一次方向键指令（即改变一次移动方向或开始移动）记为一步（按下方向键后角色连续移动直到松开或遇阻，只计 1 步）。
- 显示面板需实时刷新上述数值。

### 2.4 状态保存与恢复
- 将当前迷宫参数（`rows`、`cols`、随机种子`seed`、角色当前位置坐标）持久化到本地存储（`localStorage` 或 Electron 的 `userData` 文件），应用启动时自动恢复上次迷宫状态。
- 保存用户偏好设置（如移动速度、主题颜色等）。
- 提供“重置为默认迷宫”的选项。

### 2.5 UI 布局与交互
- 整体布局：**单窗口、居中内容**。左侧为迷宫画布区域，右侧为控制面板。
- 控制面板包含：
  - 当前状态显示（计时、步数、长度）
  - 重新生成按钮
  - 迷宫参数调整控件（行数、列数滑动条或输入框）
  - 主题/偏好设置（可选）
- 所有界面元素必须响应式，窗口默认大小 1200×800，最小 900×600。
- 无传统标题栏（使用 Electron 自定义标题栏或完全无边框，仅保留关闭/最小化等基本控制，需与整体风格融合）。

## 3. UI/UX 设计指南

### 3.1 设计原则
- **极简主义**：去掉一切非必要装饰，仅保留核心信息。
- **暗色主题**：主背景深灰/黑色（`#1a1a2e` 或类似），迷宫墙壁为浅色高亮（`#e0e0e0` 带微光），角色为纯黑或主题强调色。
- **玻璃态面板**：右侧控制面板使用半透明背景+模糊效果（`backdrop-filter: blur(20px)`），圆角，细边框。
- **微交互**：按钮悬停有柔和的背景色变化；移动角色时墙壁附近有微弱发光反馈；到达出口时画布有短暂脉冲动画。
- **字体**：统一使用系统原生等宽/无衬线字体（`'Inter', 'SF Pro', system-ui`），数字部分等宽以保持计时器整洁。

### 3.3 色彩方案

一下为初始参考，实际可根据美观程度自行开发，要支持不同主题颜色（比如配都使用变量，提供不同主题配置后切换可以一键修改）

- 背景深色：`#0f0f1a`
- 面板背景：`rgba(255,255,255,0.05)` + 模糊
- 墙壁线条：`#e2e8f0`，发光效果 `filter: drop-shadow(0 0 2px #6366f1)`
- 角色：`#ffffff` 外发光
- 强调色（按钮、链接）：`#818cf8`
- 文本：`#f1f5f9` 主文字，`#94a3b8` 次要文字

### 3.4 动画规范
- 角色移动：线性过渡，无缓动（保持速度恒定），帧率 60fps。
- 按钮/面板进入：轻微淡入+上移，持续 0.2s。
- 胜利效果：画布边缘出现脉冲光环，持续 1s，然后显示结果。

## 4. 技术架构

### 4.1 整体结构
```
maze/
├── electron/            # Electron 主进程代码
│   ├── main.ts
│   └── preload.ts
├── src/                 # Vue 前端 (渲染进程)
│   ├── App.vue
│   ├── main.ts
│   ├── components/
│   ├── views/
│   ├── store/           # Pinia 状态管理
│   ├── utils/
│   └── assets/
├── package.json
└── ...
```

- **主进程**：负责窗口创建、无边框窗口、系统托盘（可选）、文件保存等系统级操作。
- **预加载脚本**：暴露安全的 IPC 接口给渲染进程（如保存/读取用户设置到文件系统）。
- **渲染进程**：Vue 3 + TypeScript，使用 `primevue` 作为 UI 组件库（按需引入），所有核心逻辑（迷宫生成、动画、状态管理）均在此层实现。

### 4.2 关键依赖
- `electron` 
- `vue` ，`vite` 
- `primevue` （尽量都使用该三方库的组件，需要调整的则通过封装它的组件来修改css、布局和逻辑等，没有我的允许不能创建新的组件，需要创建则提前说明）
- `pinia` 用于状态管理
- `canvas` API 绘制迷宫和角色（不依赖第三方绘图库，保持轻量）
- `electron-store` (可选) 用于持久化用户设置

**注意**：不要引入大型动画库，所有动画通过原生 Canvas 和 CSS 实现。

## 5. 功能模块详细设计

### 5.1 迷宫生成算法（优先使用三方库，没有的情况再自研）
**要求：最好的性能、随机性、有效性、正确性**

**选择**：**递归回溯（Recursive Backtracker）** 算法生成完美迷宫（所有单元格连通，无环，通路唯一）。  
步骤：

1. 将整个网格视为单元格矩阵，每个单元格初始有四面墙壁。
2. 从入口单元格开始，随机选择未访问的相邻单元格，打通之间的墙壁，递归深入，直至所有单元格访问完毕。
3. 生成后，移除入口和出口对应的墙壁，标识出入口（例如入口左侧墙壁打通，出口右侧墙壁打通）。
4. 使用一个可配置的**随机种子**（`seed`），保证相同种子、相同行列数生成的迷宫完全一致，便于恢复状态。

**数据结构**：
```ts
interface Cell {
  row: number;
  col: number;
  walls: { top: boolean; right: boolean; bottom: boolean; left: boolean };
  visited: boolean;
}
```
整体迷宫用二维数组 `Cell[][]` 表示。

**复杂度参数**：暂时通过改变行列数控制难度，复杂度的进一步调整可后期通过“拆墙”概率引入环路（非 MVP）。

### 5.2 迷宫绘制（Canvas）
- 画布尺寸根据窗口大小动态计算，保持迷宫居中，单元格为正方形。
- 墙壁绘制：遍历所有单元格，根据 `walls` 属性绘制线条（`moveTo`/`lineTo`），线条宽度 2px，颜色见色彩方案。
- 角色绘制：记录角色当前像素坐标 `(x, y)`，绘制实心圆。
- 出口/入口：入口区域高亮绿色微光，出口区域高亮红色微光。

### 5.3 角色移动与碰撞检测
- 采用**网格坐标 + 像素坐标**结合方式：
  - 角色当前所在单元格为 `(gridRow, gridCol)`，像素坐标对应单元格中心。
  - 按下方向键后，检查目标方向的墙壁是否存在，若不存在，则设定目标像素坐标（相邻单元格中心）。
  - 在动画循环中，每帧将角色从当前位置向目标位置移动固定像素步长（`speed * deltaTime`），直到到达或超出目标。
  - 移动过程中持续检测是否已到达目标单元格，若到达则停止等待下一次输入。
- 行走长度累加：每帧计算 `Math.hypot(dx, dy)` 并累加。
- 步数：每次接收到新的方向指令（且成功开始移动）时 `stepCount++`。

### 5.4 状态管理 (Pinia Store)
定义一个 `mazeStore`：
- 状态：
  - `rows`, `cols` (number)
  - `seed` (number | null)
  - `grid: Cell[][]`
  - `playerPos: { x: number, y: number }` （像素坐标）
  - `playerGrid: { row: number, col: number }`
  - `isMoving: boolean`
  - `timerRunning: boolean`
  - `elapsedTime: number` (毫秒)
  - `totalDistance: number`
  - `stepCount: number`
- 动作：
  - `generateMaze(rows?, cols?, seed?)`
  - `movePlayer(direction: 'up'|'down'|'left'|'right')`
  - `startTimer() / stopTimer()`
  - `resetAll()`
  - `saveState() / loadState()`

### 5.5 数据持久化
使用 `localStorage`（渲染进程）或通过 IPC 保存至 `userData` 目录下的 JSON 文件。
存储结构：
```json
{
  "mazeParams": {
    "rows": 15,
    "cols": 15,
    "seed": 123456789
  },
  "playerGridPos": { "row": 0, "col": 0 },
  "stats": {
    "elapsed": 0,
    "distance": 0,
    "steps": 0
  },
  "settings": {
    "speed": 3,
    "theme": "dark"
  }
}
```
应用启动时尝试读取，若存在且参数有效则恢复迷宫及角色位置（角色需重新计算像素坐标），否则生成默认迷宫。

## 6. 开发注意事项（AI 开发者必读）

### 6.1 通用原则
- **提问优先**：对任何未明确的需求（例如：移动中再次按键是否中断当前移动？迷宫复杂度如何量化？），必须先向用户提问，不得自行决定。
- **最小可行实现**：先构建核心循环（生成迷宫→移动→统计→保存），再添加优化和美化，禁止一开始就设计庞大架构。
- **简洁代码**：单个函数不超过 50 行，避免过度抽象；PrimeVue 组件仅按需引入，不可全局注册整个库。
- **类型安全**：充分利用 TypeScript，禁止使用 `any` 除非绝对必要。
- **电子与 Web 分离**：渲染进程不得直接调用 Node.js API，所有系统操作通过 `contextBridge` 暴露。

## 8. 验收标准
- 应用启动，展示一个随机的线条迷宫，黑点位于入口。
- 方向键控制黑点在通道内平滑移动，无法穿墙。
- 右侧面板实时更新耗时（0.1s 精度）、步数、总行走像素长度。
- 关闭应用重新打开，迷宫和角色位置与上次一致。
- 点击“重新生成”，参数不变时产生新迷宫（或同种子产生相同迷宫），统计数据清零。
- 窗口缩放时迷宫等比缩放且居中，角色位置正确。
